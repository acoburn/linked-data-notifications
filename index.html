Abstract

Notifications are sent over the Web for a variety of purposes, including social applications. The data contained within a notification is arbitrary, and typically only usable by the application which generated it in the first place. In the spirit of Linked Data, we propose that notifications should be reusable by multiple applications. This provides end users more freedom to switch between the online tools they use, as well as generating greater value when notifications from different sources can be used in combination. Through separating out the concepts of senders, receivers and consumers of notifications, and leveraging Linked Data principles of shared vocabularies and URIs for resources, the Linked Data Notifications (LDN) protocol meets this goal. In this article we describe the protocol, which is a Candidate Recommendation of the W3C (https://www.w3.org/TR/ldn/), situate it alongside related initiatives, and discuss additional considerations arising from architectures such as this which decouple applications from data storage. We evaluate the protocol's effectiveness through multiple, independent implementations which pass a suite of formal tests and can be demonstrated interoperating with each other. Introduction

Notifications are sent over the Web for a variety of purposes, including social applications. put some specific, vivid examples here The notification data may be displayed to a human to acknowledge, or used to trigger some other application-specific process (or both). This data is arbitrary and typically only usable by the application which generated it in the first place. Current efforts towards re-decentralising the Web [cite, cite] are moving towards architectures in which data storage is decoupled from application logic, freeing end users to switch between applications, or to let multiple applications operate over the same data. So far, notifications are considered to be ephemeral resources which may disappear after transport, and thus are excluded from being reusable data.

We contend that notification data should no more be locked into particular systems than other user data, and designed the protocol Linked Data Notifications (LDN) to support sharing and reuse of notifications across applications, regardless of how they were generated or what their contents are. Specifying this architecture as a formal protocol is intended to allow applications which generate and make use of notifications, which are independently implemented and run on different technology stacks, to seamlessly work together. LDN contributes to decentralisation of our interactions on the Web, as well as encouraging generation and consumption of Linked Data.

We build on existing W3C standards and Linked Data principles. In particular, storage of notifications is compatible with Linked Data Platform servers, notifications are identified by HTTP URIs, and notification contents must be available as JSON-LD (and optionally negotiated for the other formats). LDN is a Candidate Recommendation of the W3C, and incorporated in ActivityPub, a W3C Candidate Recommendation for decentralised social networking protocol based upon the ActivityStreams 2.0 data format.

In this article we present user cases for decentralised notifications from the Social Web domain, and describe the requirements which guided the development of the protocol. We discuss related work in terms of predecessors to this work, comparisons with current alternative approaches, as well as complementary protocols which can work alongside LDN. We summarise the protocol itself, and then go into detail about specific architectural considerations that were made. We built a test suite which can be used to confirm that implementations conform with the specification, and we describe the independent implementations which interoperate with each other so far.

We use some terms throughout which may be subject to different interpretations by different communities, so we provide some definitions here.

Decentralisation
Data and applications are loosely coupled, and users are empowered to choose where their data is stored or held. We focus on Web-based decentralisation, where content is transported over HTTP, and resources are identified with URIs.
Inbox
A container or directory which is used to store and serve a collection of notifications.
Notification
A retreivable resource which returns RDF. The contents of notifications are intended to describe a change in state of some other resource, or contain new information for the attention of a user or process, and may be subject to constraints of the Inbox in which it is contained.

The Webmention protocol is used to notify that content at the source URL includes the target URL string to the receiver of a Webmention endpoint.

The sender MUST post x-www-form-urlencoded [HTML5] source and target parameters to the Webmention endpoint, where source is the URL of the sender's page containing a link, and target is the URL of the page being linked to.

Webmention does not define the notion of an entity that is accessible at an URL for the mention itself. The mention is only intended for the receiver to process. There are no syntactical restrictions set on the content of the source URL, e.g., it can be plain-text, HTML, JSON, or a binary file, therefore it is effectively foremost about the source URL containing the target URL as a string.

A Webmention workflow is as follows: 1) Alice has an article in-reply-to Bob's article, 2) Bob's Webmention endpoint is discovered to deliver a mention, 3) a Webmention is sent to Bob's endpoint indicating that Alice's mentions Bob's article.

Scope and Approach: The source URL is the most essential component since it identifies the origin of the claim and where to locate it. Since the information that is provided in the POST is only a claim, it may or may not be of interest to the receiver, and the receiver makes no promise to do anything with it e.g., displaying.

The target URL is used for the purpose of optimisation, i.e., making it easier for the recipient to find the target URL string at the source URL. The target URL is also used to allow external (third-party) services to help site owners without a Webmention endpoint.

Analysis: Verification of Webmention claims in source formats that's in plain-text or binary are ambiguous due to unspecified characteristic of where an URL string starts and ends. Hence, exact matching the string URL is not deterministic as there may be partial matches which are false positives. For structured formats like HTML and JSON, it has inherent limitations given that:

  • While the content at source may indicate (in any convention or serialisation format) the type of relation between the source and target URLs, the information about the relation is not transmitted to the receiver's endpoint. Hence, the receiver implementation interprets the relation (or lack of) between the source and the target. The untyped relation is interpreted as a generic mention by default.
  • If the source URL contains multiple mentions of the target URL, the Webmention endpoint is not informed about the particular instance. Therefore, the receiver is required to process the whole document and base its verification on the existence of one of those mentions. This sets forth an indeterministic verification step since there is no clear way to distinguish the received Webmention at the endpoint from multiple mentions which can exist at the source.

Requirements and Design Considerations

In this section we discuss some considerations for a Web notification protocol that conforms to the Linked Data design principles, as well as best practices for applications. We use the considerations towards establishing normative requirements for the protocol; what to be expected from the core for interoperable application behaviour, as well as notification data exchange. The LDN protocol will refer to existing specifications in favour of keeping the protocol design simple and modular for reuse.

Architectural Considerations

As Linked Data encourages the use of Representational State Transfer (REST) architectural style [] also Linked Data Notifications should be compatible with REST to the largest extent possible. This also ensures interoperability with other standards such as for example Linked Data Platform (LDP). A RESTful realisation also has further advantages, such as organisation of atomic resources, including identification of units of information, their discovery and description, as well as basic operations like CRUD on individual resources through HTTP methods.

Sender Verification

In order to establish authenticity of information receivers should verify the sender of a notification. This can be done for example using whitelists, requiring authentication, verification through dereferencing an authoritative resource, or checking a digital signature.

For abuse prevention, servers receiving the notifications should set and advertise constraints to filter unwarranted notifications from being created on the server and exposed for consumption.

Notification Verification and Validity

Verification: If a notification refers to or describes third-party (external) resources, fetching and processing their descriptions may be one way to verify their existence.

Validation: Given the nature of the notification payload, consumers may want to take precautions when ascertaining the veracity of the contents. The anyone can say anything about anything Web principle applies here. Unless otherwise specified (e.g., through HTTP headers, or updating the notification content), there may be no indication as to whether the original payload was altered by the receiver.

Notification Constraints

Setting constraints on notification data shape is motivated with the purpose of attaining application and data integrity. For application integrity, some of the common reasons for this is to minimise maintenance complexity, and to generate conforming data for optimal reuse. For data integrity, constraints makes it possible to store and serve data as intended, as well as optimal reuse by different conforming applications, and play a key role in accuracy and consistency in data's life-cycle.

If the notification constraints are advertised as structured descriptions, generation and consumption of the notifications can be better automated by applications. Inboxes could announce their own constraints using existing data shapes like with SHACL or other recommendations.

Pushing and Pulling Notifications

Applications can either ask to have incoming notifications be pushed to them or they can actively pull notifications from servers. Both of the mechanisms are specified with existing implementations in the wild. Typically where one is preferable over another depends on application environments and the type of arrangement between senders, receivers, and consumers.

Linked Data Notifications (LDN) is a protocol that describes how servers (receivers) can have messages pushed to them by applications (senders), as well as how other applications (consumers) may retrieve those messages. Any resource can advertise a receiving endpoint (Inbox) for the messages. Messages are expressed in RDF, and can contain any data.

LDN is a specialized use of Linked Data Platform [LDP] for sending and consuming notifications. It is not dependent on a complete implementation of LDP, but an easy-to-implement subset. LDN is a W3C Recommendation [LDN]

The LDN protocol supports a modular approach to creating and storing data, in particular a decoupling between applications and data storage. Thus, where senders, receivers and consumers of notifications are independently implemented and run on different technology stacks, this protocol allows them to seamlessly work together.

  • This specification enables the notion of a notification as an individual entity with its own URI which can be retrieved and reused.
  • This specification deliberately does not define the vocabulary of the notification contents in order to allow for use in a range of different application domains.
  • Authentication and verification of notifications is encouraged, but the mechanism to do so is at the discretion of receivers and consumers, as needs differ according to types of notification and different application domains.

Overview

A sender is triggered, either by a human or an automatic process, to deliver a notification to a server. The notification is data intended for the attention of the receiver, for example: a personal message from a friend; a pingback link; a comment on a blog post; an invitation to collaborate; a calendar reminder; a scientific observation.

The sender chooses a target resource to send notifications to; the sender then discovers the location of the target‚Äôs Inbox, and sends the notification there. Any resource can advertise an Inbox. The receiver exposes the notification data (according to appropriate access control) for use by consumers.

Consumers discover the location of the Inbox in the same way as the sender, and may perform further processing on the notifications, combine it with some other data, or simply present it in a suitable human-readable way.

Summary

Senders and consumers discover a resource‚Äôs Inbox URL through a relation in the HTTP Link header or body of the resource.

The Sender:

  • Creates the body of the notification according to the needs of application.
  • Sends the notification to the Inbox URL by making a POST request, containing the body in JSON-LD or in another serialization acceptable by the server.

The Receiver:

  • Responds to GET requests made to the Inbox URL with a listing of the URLs of notifications that have previously been accepted.
  • Responds to GET requests made to the individual notification URLs with JSON-LD (or optionally other serializations).
  • Accepts POST requests at the Inbox URL to create notifications.
  • Optionally enforces constraints on notifications sent to the Inbox.

The Consumer:

  • Retrieves the contents of the Inbox URL with a GET request, and uses according to the needs of application.

Data Formats

While the ability for applications to generate payload, conduct content negotiation, and consume content in all W3C RDF serializations, has advantages to be flexible, it can be costly to support across applications. From the perspective of the protocol's design, there are some considerations to take into account, for example:

  • application interoperability
  • maintenance of the RDF parsers and serialisation libraries
  • complexity of their inclusion in applications
  • run-time efficiency

With the goal of enabling simple yet interoperable application creation, LDN requires applications to speak and understand the JSON-LD syntax ‚Äî for looking at the contents of Inbox as well as individual notifications. Applications are permitted to optionally negotiate different RDF serialisations among themselves.

The rationale for this is that, if all applications can work with one serialisation, then they are interoperable. Applications that are capable of or wish to handle different serialisations are able to coexist in this ecosystem nevertheless. The benefit to making JSON-LD is that it provides the lowest barrier for applications to communicate. While the same can be said for any RDF serialisation, JSON-LD is arguably more convenient to work with than the other serialisations in client-side implementations.

Target ownership: Publishers of the resources advertising an Inbox (target) should do so on a server they trust. Publishers must be aware that third-party access to headers or content could result in notifications being redirected.

Natural Language Content: Building an international base of users is important in a federated network. Implementations are encouraged to provide means of discovering the available languages and/or negotiating the language returned, such as using the HTTP Accept-Language header to negotiate and select the most appropriate language representation to send for a given request.
